Many PC-oriented table components have a large number of columns. When viewed on a mobile device in portrait orientation, the horizontal space is insufficient, and users must scroll extensively to see all the data. Rotating to landscape orientation effectively swaps the screen's width and height, providing a much wider viewport for table content.

## Why CSS `transform: rotate()` Instead of Alternatives?

There are several ways to achieve a landscape layout on mobile, but each has significant trade-offs:

| Approach | Drawback |
|----------|----------|
| **`screen.orientation.lock('landscape')`** | Requires the page to be in fullscreen mode first (`Element.requestFullscreen()`). Most browsers reject the lock outside of fullscreen or user-initiated events, and iOS Safari does not support it at all |
| **Force users to rotate the device** | Relies on the user's physical action and the device's auto-rotate setting. Cannot be triggered programmatically, and many users keep rotation lock enabled |
| **Responsive redesign** | Requires completely redesigning the table layout for narrow screens — high cost, and may not suit data-dense tables that inherently need many columns |

**CSS `transform: rotate(90deg)`** avoids all of the above problems:

- Works on **all browsers and all devices** — no API restrictions, no permission prompts
- Is **purely programmatic** — triggered by a button tap, no reliance on device orientation
- **Preserves the original table layout** — no need to redesign columns or break the PC table structure
- Has **zero runtime cost** — GPU-composited transform, no reflow or repaint

The only downside is that after CSS rotation, the physical touch axes and the element's logical scroll axes become misaligned. That is exactly what the `rotation` parameter in `v-table-touch-scroll` solves — it automatically applies an inverse coordinate transform to correct the touch mapping.

## How It Works

1. Use CSS `transform: rotate(90deg)` to rotate the table container
2. Swap width and height (`width: 100vh`, `height: 100vw`) so the rotated container fills the screen
3. Set `rotation: 90` on the directive — internally, `transformCoords()` maps touch coordinates `(clientX, clientY)` to `(clientY, -clientX)`, aligning scroll direction with the rotated table's logical axes

## Configuration

The `rotation` parameter accepts the following values:

| Value | Description |
|-------|-------------|
| `0` | No rotation (default) |
| `90` | Container rotated 90° clockwise |
| `-90` | Container rotated 90° counter-clockwise |
| `180` | Container rotated 180° |

```vue
<template>
  <div v-table-touch-scroll="{ preset: 'element-plus', rotation: 90 }">
    <el-table :data="tableData" height="100%">
      <!-- columns -->
    </el-table>
  </div>
</template>
```

## Transform Origin

The combination of `transform-origin: top left` and `translate: 100vw 0` is the key positioning technique. After rotating 90° clockwise around the top-left corner, the container shifts off-screen to the left. The `translate: 100vw 0` pushes it back, making it cover the entire viewport precisely.

## Notes

### Safe Area Insets

On devices with notches or rounded corners (e.g., iPhone), make sure to account for safe area insets:

```css
.table-container.is-landscape {
  padding: env(safe-area-inset-top) env(safe-area-inset-right)
           env(safe-area-inset-bottom) env(safe-area-inset-left);
}
```

### Responsive Sizing

When your project uses a framework with built-in responsive font scaling (e.g., font size automatically increases on larger screens), rotating to landscape mode triggers this scaling. However, if table column widths are fixed in pixels, they won't scale proportionally, causing column headers to wrap or content to overflow.

**Example scenario:**
- Portrait mode: font-size = 14px, column width = 100px (looks balanced)
- Landscape mode: font-size = 18px (auto-scaled), column width = 100px (unchanged) → header text wraps

**Solution:** Use a helper function that calculates column width based on both viewport width and a maximum constraint:

```typescript
// Remove this comment line when using in your project
// eslint-disable-next-line no-restricted-imports
import { round } from "lodash-es"

// Design reference width (mobile ideal width)
const idealWidth = 375

// Maximum width for larger screens
const maxWidth = 600

/**
 * Calculate responsive width that scales with viewport but respects a max constraint
 * @param {number} n - Base width in pixels (at ideal width)
 * @returns {string} CSS min/max function string
 */
export function vw(n: number) {
  if (n === 0) return n

  const vwN = round((n * 100) / idealWidth, 3)
  const maxN = round((n * maxWidth) / idealWidth, 3)
  const cssF = n > 0 ? "min" : "max"
  return `${cssF}(${vwN}vw, ${maxN}px)`
}
```

Apply the `vw()` function to your table column widths:

```vue
<script setup lang="ts">
import { vw } from './utils/vw'

const columns = [
  { prop: 'id', label: 'ID', width: vw(60) },
  { prop: 'name', label: 'Name', width: vw(120) },
  { prop: 'email', label: 'Email', width: vw(200) },
  { prop: 'phone', label: 'Phone', width: vw(150) },
]
</script>

<template>
  <div v-table-touch-scroll="{ preset: 'element-plus', rotation: isLandscape ? 90 : 0 }">
    <el-table :data="tableData" height="100%">
      <el-table-column
        v-for="col in columns"
        :key="col.prop"
        :prop="col.prop"
        :label="col.label"
        :width="col.width"
      />
    </el-table>
  </div>
</template>
```

This approach ensures that as the viewport (and font size) grows, column widths scale proportionally, maintaining visual balance across all screen sizes.
